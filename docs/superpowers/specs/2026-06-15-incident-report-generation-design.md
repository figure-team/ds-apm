# 장애보고서 자동 생성 (설계 노트 · 추후 작업)

- 상태: **Proposed (미착수)**
- 작성일: 2026-06-15
- 관련: CF-2(AI 1차 분석 초안), CF-11(코드 RCA), incident 라이프사이클
- 형제 노트: `2026-06-15-cf2-comms-template-grounding-design.md` (B, comms 템플릿 그라운딩)

## 배경 / 동기

한국 SI·엔터프라이즈 사고관리는 실시간 대응으로 끝나지 않는다. 복구 후 **공식 장애보고서**를 요구하며, 이는 흔히 **계약·SLA·감사** 요건이라 "안 주면 안 되는" 산출물이다. CF-2가 현재 만드는 것은 *발생 중 실시간 대응 초안*(헤드라인·조치·고객/벤더 공지)이고, 장애보고서는 결이 다른 **사후 완성 문서**다.

> 출처 통찰(2026-06-15): "한국 SI는 쉽지 않다. 장애보고서를 원할지도 모른다." + "한국 software 시장은 문서에 목맨다."

## 핵심 통찰: "집약 산출물"

새로 다 만드는 게 아니라, **이미 가진 조각을 한 문서로 합치는** 것이 본질이다.

| 보고서 섹션 | 출처 |
|---|---|
| 장애 개요·등급 | 알람(CF-2 입력) |
| 발생/인지/복구 일시, 소요시간 | 알람 타임스탬프 + incident 라이프사이클 |
| 영향 범위 | 알람 라벨 + evidence |
| **장애 경과(타임라인)** | 조치 이력(firstActions 실행 기록 + 핸드오프 sink) |
| **원인 분석(직접/근본)** | **CF-11 코드 RCA** + CF-2 가설 |
| 조치 내역 | firstActions + 실제 수행 로그 |
| **재발 방지 대책** | CF-11 proposedFix + 신규 생성 |
| 고객 영향/공지 | CF-2 customerUpdateDraft(공지문 양식) |

즉 **장애보고서 = CF-2(대응) + CF-11(근본원인) + incident 타임라인** 을 한국 SI 양식 문서로 묶는 것. 그라운딩·공지문 양식·문서 포맷 노하우가 그대로 재사용된다.

## CF-2/11과 결이 다른 점 (별도 CF일 수 있는 이유)

- **시점**: CF-2/11은 *발생 중*, 보고서는 *복구 후*(또는 진행 중 중간보고)
- **출력**: 초안 필드(문자열) → **완성 문서**(표·타임라인·다중 섹션; 마크다운→PDF/HWP)
- **승인/이력**: 결재 라인 + 버전 관리 필요(초안과 다른 거버넌스)

## 의존성 (선결 과제)

- **incident 타임라인·조치 실행 기록의 영속화**가 보고서 "경과/조치 내역" 섹션의 핵심. 현재 이 데이터를 저장·조회하는지 확인 필요(AI strategy history는 있으나 조치 *실행* 타임라인은 별개).
- CF-11 run 결과(근본원인·proposedFix)와 CF-2 strategy를 incident 단위로 **연결(correlate)** 하는 키 정리.

## 방향 / 오픈 퀘스천

- 새 CF로 분리(예: "CF-X 장애보고서 자동생성") vs CF-2 확장 — 시점·출력·거버넌스가 달라 **분리 쪽이 자연스러움**.
- 양식: 고객사마다 다른 보고서 템플릿 → **B의 템플릿 그라운딩 패턴 재사용**(SOP/org 스코프 템플릿에 슬롯, LLM이 채움).
- 출력 포맷: 마크다운 1차 → PDF/HWP 변환, 타임라인 표 렌더.
- 중간보고 vs 최종보고 2단계 산출 지원?
- 보상/법적 책임 섹션은 가드(자동 단정 금지) 유지 — comms 가드와 동일 정책.

## 범위 / 비용

타임라인 영속화(선결) + 집약 생성 + 템플릿 그라운딩 + 문서 변환. **큰 작업**(여러 CF 횡단). 단, 조각(CF-2/11)이 이미 있어 신규 생성 부담은 "집약·문서화"에 집중됨. 우선순위는 시장 요구(SLA 계약 여부)에 따라.
