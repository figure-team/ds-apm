## 3. 대기 중인 결정 (다음 세션 핵심)

### 3.1 ★ SSG(정적 사이트 생성기) 도입 — 결정 보류, "더 비교" 요청됨
**배경**: 상세본을 md-only로 만들었더니 mermaid 다이어그램이 raw 텍스트로만 보임 → "HTML 이점을 못 살린다". 근본 원인은 **self-contained 수제 HTML이 잘못된 축**이었다는 것. 웹 리서치 2건으로 검증 완료:

- **"single-file HTML이 좋다"(thariqs html-effectiveness)는 일회성 시각 산출물 전달에 한정된 주장.** 지속 편집 + mermaid 다수 + LLM 소비 + 한국어 장문 spec에는 부적합. 우리가 over-apply한 것.
- **업계 LLM-first 합의 = `md=source → SSG가 HTML 빌드`**: 사람은 렌더 HTML, AI는 같은 URL raw `.md`. mermaid는 **빌드타임 SVG pre-render**(CDN 의존 0 + 렌더 품질 + md엔 텍스트 남아 LLM이 읽음). llms.txt 자동 생성.

**SSG 후보 비교** (다음 세션이 결정):

| 도구 | mermaid | 빌드 | 한국어검색 | LLM(llms.txt) | 환경 |
|---|---|---|---|---|---|
| **MkDocs Material** (1위 추천) | ✓ 네이티브 1줄 | ✓ pip 1개, 초단순 | △ Pagefind 보강 | △ 서드파티 | Python (현 http.server와 동일) |
| **Astro Starlight** (2위) | △ 서드파티 클라렌더 | △ Node 무거움 | ✓ Pagefind 내장 최강 | ✓ 공식 starlight-llms-txt | Node |
| VitePress / Docusaurus / Hugo | — | — | — | — | (리서치 결과 우위 없음/과함) |

**미해결 하위 결정**:
1. 어떤 SSG (MkDocs Material vs Starlight) — 사용자가 "더 비교" 요청. 샘플 빌드로 눈으로 보고 싶어함.
2. 적용 범위: 전체 SSG 통합(요약본 self-contained HTML도 흡수) vs 상세본만 SSG + 요약본 HTML 유지
3. 결정 후 `PROCESS.md §8`을 SSG 빌드 정책으로 재개정 (현재 "상세본 md-only / 요약본 self-contained HTML"로 박혀 있는데, SSG 도입 시 이 전제가 또 바뀜)

> 출처: SSG 비교 — squidfunk.github.io/mkdocs-material(diagrams,search), starlight.astro.build(site-search), llmstxt.org, youngju.dev 2026 SSG 비교. HTML/md 포맷 — thariqs.github.io/html-effectiveness(원문), beam.ai·releasepad·tarikdavis(md vs html for agents), Anthropic docs/llms.txt 실물, Mintlify ai-native.
