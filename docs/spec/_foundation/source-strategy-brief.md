## 1. 추진 배경 및 문제점
- **고비용 인력의 저부가가치 업무 점유**: 연간 대형 장애는 2~3건에 불과하나, 상시 모니터링·데이터 수집에 PM/업무 PL 리소스가 매일 1~2시간씩 고정 낭비.
- **분석·증명 리소스 과다**: 인시던트(연간 약 10건) 발생 시 원인 분석·보고서 작성을 위해 PM/PL이 직접 트랜잭션·로그 대조 등 단순 확인에 고단가 리소스 투입.
- **운영 조직의 고착화**: 특정 인력 의존도가 높아 조직 유연성·대응력 저하, 인력 변동 시 대처 어려움.
- **지식의 개인 편중**: 특정 PM/PL 경험 의존 → 담당자 부재 시 자생적 대응력 현저히 낮음.

## 2. 사업 추진 목적 및 최종 목표
- **PM의 역할 확장 (To-AM)**: 기술적 운영(Ops)을 AI·시스템에 이관, PM을 고객 가치 제안·비즈니스 확장의 AM(Account Manager)으로 전환.
- **운영 조직의 상향 평준화**: SOP 기반 조치 가이드로 **초/중급 인력만으로 인시던트 1차 대응** 가능한 체계 구축.
- **기술 자산화**: 자동 보고서 생성 + 스펙-코드 체인 연결로 운영 지식의 시스템 내재화.

## 3. 핵심 추진 전략
**기술 전략**
- AI-Ops 자동화 도입: Runbook/SOP 기반 인시던트 탐지·분류·보고 자동화
- SigNoz(OSS) 기반 모니터링 통합: 실시간 장애 감지·로그 분석
- 지식 시스템 구축(LLM Wiki): 자동 보고서 생성 + 스펙·코드 연동

**조직 전략**
- PM 역할 재정의(Escalation & RCA): 단순 Level 1 기반 기술 운영(Ops) 업무를 AI 시스템으로 이관
- AM 전환(역할 확대): 고객 가치 제안·비즈니스 확장 중심으로 역할 이동
- 운영 표준화(자산화): SOP 기반 재기동 자동화(Auto-Remediation) + 비이상적 임계치 사전 경고

## 7. 결론
- 관제–지식화–ITSM 통합으로 운영 경쟁력을 내재화하는 Lock-in 효과 확보
- AI-Ops는 단순 기술 혁신이 아닌 **운영 체계 혁신**
- 완전 자동화가 아닌 **지능형 자동화 + 효율 극대화**
- **데이터 중심 운영 문화 구축**이 핵심 성공 요인

## 별첨 2 — 소규모 프로젝트 Observability 도입 검토 보고

**결론**: OSS 기준 1순위는 **SigNoz로 PoC 진행**. 빠른 설치는 ClickStack, 로그 중심은 OpenObserve, 운영 부담이 크면 WhaTap 유료 전환 검토.

**1. 목적**: 소규모 프로젝트에서 장애 원인을 빠르게 확인하기 위해 로그·인프라 상태·대시보드를 제공하는 최적 관제 도구 검토.

**2. 비교표**

| 구분 | SigNoz | ClickStack / HyperDX | OpenObserve | ELK | WhaTap 유료 |
|---|---|---|---|---|---|
| 형태 | OSS / Self-host | OSS / Self-host | OSS / Self-host | Self-host 중심 | 유료 SaaS |
| 필수 기능 | 로그·인프라·대시보드 | 로그·인프라·대시보드 | 로그·인프라·대시보드 | 가능하나 구성 복잡 | 로그·인프라·대시보드 |
| 알림 | 가능 | 가능 | 가능 | 가능 | 가능 |
| 설치 난이도 | 중간 | 낮음 | 낮음~중간 | 높음 | 낮음 |
| 운영 부담 | 직접 운영 | 직접 운영 | 직접 운영 | 높음 | 낮음 |
| 비용 구조 | Self-host 클라우드 비용 | Self-host 클라우드 비용 | Self-host 클라우드 비용 | Self-host 클라우드 비용 | 구독/사용량 기반 |
| 종합 판단 | **1순위 추천** | 빠른 PoC 대안 | 로그 중심 대안 | 소규모엔 다소 무거움 | 운영 편의성 우수 |

**3. 선정 이유 및 상세**
- **SigNoz**: 로그·인프라·대시보드·알림 균형 제공 → 1순위 OSS 후보.
- **ClickStack**: All-in-one Docker로 가장 빠른 PoC. 운영 시 저장/보관 정책 검토 필요.
- **OpenObserve**: 로그 검색·비용 효율 중시 시 대안. 라이선스/운영 방식 사전 확인.
- **ELK**: 대규모 로그 분석 강점이나 구성요소 많아 소규모 빠른 적용엔 무거움.
- **WhaTap**: 유료 SaaS, 운영 부담 낮고 지원 좋으나 사용량/대상별 비용 발생.
- **PoC 범위**: 로그 검색, 서버/컨테이너 CPU·메모리 확인, 비개발자용 기본 대시보드.
- **비용 해석**: OSS는 라이선스 비용 낮으나 클라우드 서버·스토리지·백업·운영 인력 비용 발생.
- **Uptime Kuma**: APM/Observability 도구의 알림 기능으로 우선 처리. 외부 생존 체크·고객용 상태 페이지 필요 시에만 추가.
- **최종**: SigNoz로 PoC 시작, 설치 속도 최우선이면 ClickStack, 운영 부담 크면 WhaTap 전환 검토.
