# Story 11.3: 폭주 비용·볼륨 제어 (admission·lease)

Status: done

## Story
**운영 조직**은 동일 오류 수백 건이 동시에 떠도 분석 실행 수가 강제 제한되어 토큰·프로세스가 폭주하지 않음을 보장받는다. *(목적: quota 폭주 이력(y2i)을 반복하지 않도록 — #1 설계 동인)*

## Acceptance Criteria
1. 한 dedup_key에 N개의 admission이 동시에 시도돼도 정확히 1건의 run만 적재된다(나머지 skip=deduped, hit_count 증가) (FR-CF11.3)
2. dedup은 **슬라이딩 쿨다운**(기본 6h, `last_admitted_at` 비교)이다 — 버킷 경계 이중 admit 없음
3. org 일일 한도(`max_runs_per_day`, 기본 20) 초과 시 skip=budget_exhausted (조건부 카운터 증가)
4. 큐 깊이(`max_queue_depth`, 기본 50) 초과 시 skip=queue_full
5. 동시성 캡은 **잠금된 `coderca_capacity` 행**의 조건부 UPDATE로 강제된다(bare count race 금지), 기본 1·최대 2
6. lease 만료(crash 모사) run은 reaper가 requeue하고 `max_attempts`(기본 2) 이내 재실행된다
7. 재clain된 run의 원래 워커 종료 쓰기는 fencing 토큰(`WHERE id=? AND lease_token=?`)으로 거부된다
8. skip은 행 폭증 없이 카운터로 집계된다(`hit_count`/`coderca_skip_stat` per (org,reason,day)) + 감사 sampled

## Tasks / Subtasks
- [x] 원자적 admission(`RunInTxCtx`) — dedup 유니크 `(org,dedup_key)` + 슬라이딩 쿨다운 (AC: 1,2)
- [x] per-(org,day) 예산 조건부 카운터(`coderca_budget`) (AC: 3)
- [x] 큐 깊이 게이트(`coderca_run` status=queued count) (AC: 4)
- [x] 잠금 capacity 행 동시성 캡(`coderca_capacity`) + claim 트랜잭션 (AC: 5)
- [x] lease/heartbeat/reaper + fencing 토큰 + capacity 재조정 (AC: 6,7)
- [x] skip 카운터(`coderca_skip_stat`) + sampled audit (AC: 8)
- [x] **EXIT GATE**: flood-sim T1 — no double-admit, no duplicate DB finalize, 시뮬 재시작 (AC: 1,5,7)
