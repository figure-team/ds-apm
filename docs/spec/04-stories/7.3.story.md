# Story 7.3: 코드베이스 근거 원인 분석 보고서

Status: planned

## Story
**운영자**는 신규 예외에 대해 코드베이스(소스·변경이력) 근거가 포함된 원인 분석 보고서 초안과 2차 알림을 받는다. *(목적: 로그만으로는 어려운 원인 추론·초기 대응을 전문가 없이 시작하기 위해)*

## Acceptance Criteria
1. status=notified 시그니처가 비동기 워커에 의해 처리된다 (FR-CF7.3)
2. 스택트레이스에서 파일/라인이 파싱되고(Java/Go/Python/Node), 로컬 미러에서 소스 스니펫(±30라인)·blame·최근 커밋 5건이 추출된다
3. 코드 컨텍스트 총량 상한 32KB가 강제된다
4. LLM 호출은 기존 aigenerator provider 경유 — quota·사용량 이력·감사 동일 적용
5. 보고서(원인 추론·근거 커밋·초기 대응 가이드·Runbook 초안)가 `ds_exception_reports`에 저장되고 status=analyzed로 전이된다
6. 2차 알림(요약+UI 링크)이 동일 채널로 전송된다
7. 예외 샘플·보고서는 저장 전 PII 마스킹(CF-4)을 거친다

## Tasks / Subtasks
- [ ] 분석 워커 골격 (`pkg/ruler/codeanalyzer/`) — DB 큐(status 폴링) 단일 goroutine (AC: 1)
- [ ] 언어별 스택트레이스 파서 + 픽스처 테스트 (Java/Go/Python/Node) (AC: 2)
- [ ] 코드 컨텍스트 추출기 (스니펫·blame·log, 총량 상한) (AC: 2,3)
- [ ] LLM 프롬프트 구성 + aigenerator 연동 (mockaigenerator로 결정적 테스트) (AC: 4)
- [ ] 보고서 도메인 모델·store (`ds_exception_reports`, 마이그레이션 081 일부) (AC: 5)
- [ ] 2차 알림 전송 (AC: 6)
- [ ] PII 마스킹 적용 지점 (저장·알림 전) (AC: 7)
