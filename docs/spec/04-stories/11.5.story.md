# Story 11.5: 분석 기준 커밋 pin·echo

Status: done

## Story
**운영자**는 분석이 어느 기준 커밋(baseline)에 대해 수행됐는지 명시받는다. *(목적: 프로덕션이 다른 커밋을 돌고 있어도 어느 버전을 분석했는지 대조하기 위해)*

## Acceptance Criteria
1. 트리거 시점에 대상 브랜치 HEAD가 `baseline_commit`으로 고정되어 `coderca_run`에 기록된다 (FR-CF11.5)
2. 소스는 on-demand로 준비된다 — 캐시된 bare clone을 `git fetch` → 대상 브랜치 HEAD 해석 → per-run 일회용 worktree/checkout을 그 커밋에 생성
3. 서비스가 저장소에 매핑되지 않으면 실행되지 않는다 (skip=no_repo_mapping)
4. 분석 결과(RCAResult)에 분석한 기준 커밋이 에코된다
5. bare-clone fetch도 동시성 캡 아래에서 수행되어 디스크/네트워크를 묶는다

## Tasks / Subtasks
- [x] 서비스→저장소 해석기 (`(org,service)→repo[+subpath]`) (AC: 3)
- [x] 순수 소스-상태 전이(git 사실 → 다음 상태 + baseline) — `GitRunner` 인터페이스 + fake 테이블 테스트 (AC: 1,2)
- [x] 실제 git 어댑터 `ShellGitRunner`(os/exec) — fetch·HEAD 해석·checkout (AC: 2)
- [x] `RCAResult.BaselineCommit` 에코 + 파서 처리 (AC: 4)
- [x] service-map store(SQL) + RepoResolver concrete (AC: 3)
