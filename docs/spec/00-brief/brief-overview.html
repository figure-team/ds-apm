DS-APM 시스템 개요

SigNoz 알림 경로에 SOP 매칭 · AI 초안 · 실패 재처리 단계를 더한 확장 모듈 그룹. 운영자는 검토·승인만 합니다. 아래는 시스템 경계 · 구성 · 품질 목표 요지입니다.

시스템 경계

외부 HTTP는 입구 1종(OTel 텔레메트리)·출구 2종(LLM API·5채널)뿐. SigNoz Alertmanager → AIOpsAgent는 같은 프로세스 안의 함수 호출(Webhook 아님)입니다.

구성 — 6 컴포넌트

6개 모듈의 합 = AIOpsAgent 전체 (누락·중복 없음). 각 모듈은 WBS와 1:1.

통합 방식 결정 (ADR-001)

초기 Python 오케스트레이터(ds_apm_poc)+SigNoz bridge 구조는 두 런타임 운영·중복 구현·스키마 어긋남 문제가 있었습니다. 2026-05-19 결정: Python 폐기 → SigNoz Go 바이너리에 내장된 확장 모듈로 구현. 단일 운영·배포 표면 절반·5채널 자산 재사용. 상세는 ADR-001.

알려진 위험 4가지
R-1 보안HMAC 서명 정책 미정 — 재발송 위변조 검증. 추후 결정.
R-2 격리다중 테넌트 격리 production-ready 미달. 내부 단일 운영엔 영향 없음.
R-3 보안PII 마스킹 입구 1점. Collector 단 이동 검토.
R-4 안정성이미 해소 — LLM 실패 시 SOP 원문 발송(fail-open), 손실 없음.
이해관계자별 책임
운영자AI 초안 검수·승인, 발송 모니터링, 실패 시 수동 재발송
SRE시스템 이상 알람 수신 + 자격증명 회전
Platform Admin테넌트 정책 등록, AI 전략 활성화, 감사 sink 설정
SecurityPII·HMAC·테넌트 격리 정책 검토
