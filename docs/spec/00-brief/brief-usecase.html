DS-APM 실사용 시나리오

새벽 결제 5xx 장애 한 상황에서 세 가지 결말 — 정상 · 메신저 실패 · AI 실패. "누가·무엇을·어떻게"만 남겼습니다. 운영자는 어느 경우에도 판단·승인만 하고, 정보는 누락되지 않습니다.

세 가지 시나리오

시나리오한 줄약속
1 · 정상장애 → 메신저 도달 약 30초. SOP+AI 초안 함께, 운영자 승인 1회 → 5채널 동시 발송30초
2 · 메신저 실패일시 장애 시 자동 재시도 → 실패분 보관함(DLQ) 보존 → 운영자 재전송(중복 차단)무손실
3 · AI 실패LLM 인증/quota 실패 시 SOP 원문 그대로 발송. 침묵 안 함대응 지속

더 보기 (필요할 때만 펼치기)

시나리오 1 — 정상 (Golden Path)
1 알람Alertmanager 발화 → AIOpsAgent 수신. "결제 5xx 초과" + 관련 SOP + AI 3단계 초안(게이트웨이→배포 확인→롤백) 함께 도달. 위키 안 뒤짐.
2 승인운영자가 5초 검토 후 "approve" 한 번.
3 발송Slack·Teams·PagerDuty·Webhook·Email 5채널 동시 도달(유일한 외부 출구). 운영자는 곧장 1단계 진입.

    participant EXT as 외부 텔레메트리(OTLP)
    participant SYS as 시스템(SigNoz+AIOpsAgent)
    participant OP as 운영자
    participant LLM as LLM API(외부)
    participant MSG as 5채널(Slack/Teams/PD/Webhook/Email)
    EXT->>SYS: OTLP 텔레메트리
    Note over SYS: Ruler 평가 → Alertmanager 발화 → Ingress → PII 마스킹 → SOP 매칭
    SYS->>LLM: AI 초안 요청 (HTTPS)
    LLM-->>SYS: 3단계 조치 초안
    SYS->>OP: 알람 + SOP + 초안 통합 발송
    OP->>SYS: 검토 후 "승인" 한 번
    SYS->>MSG: 5채널 동시 발송 (HTTP/SMTP)
    MSG-->>SYS: 전송 완료
    Note over SYS: 감사 로그 기록

예외 · SOP 못 찾으면 원본 알람만 / PII 마스킹 실패 시 발송 중단 / 운영자 무응답(긴급 5분·일반 15분) 시 SOP 원문 자동 발송 — 침묵 안 함.

시나리오 2 — 메신저 전송 실패
재시도1→2→4초 간격 최대 3회. 일시 장애면 대부분 회복.
보관함3회 실패 시 원본·시도이력·사유를 DLQ(JSONL 파일)에 보존. 임의 폐기 없음.
알람"보관함 1건 적재" 운영자에게 알림.
재전송정상화 확인 후 "재전송" 한 번 → 중복 차단 후 발송.

    autonumber
    participant SYS as 시스템
    participant MSG as 5채널(외부)
    participant OP as 운영자
    SYS->>MSG: 발송 시도 1회차
    MSG-->>SYS: 실패 응답
    Note over SYS,MSG: 자동 재시도 3회 (1·2·4초)
    Note over SYS: 최종 실패 → DLQ(JSONL)에 적재
    SYS->>OP: "보관함 1건 적재" 알람
    OP->>SYS: 보관함 조회 + "재전송"
    Note over SYS: replay ledger 조회 → 중복 차단
    SYS->>MSG: 신규 발송
    MSG-->>SYS: 정상
    Note over SYS: DLQ "해결됨" + 재전송 이력

핵심 · 무손실 — 외부 메신저가 안 받아도 알람은 사라지지 않음. 예외 · 보관함 쓰기 실패 시 SRE critical + 메모리 버퍼 / 재전송도 실패 시 escalation.

시나리오 3 — AI가 답을 못 줄 때 (fail-open)

외부 LLM이 인증 거부(key 만료)·quota 초과로 응답을 안 주면, quota controller가 fail-open 판정 → SOP 원문 그대로 발송("AI 초안 없음" 표시). 동시에 SRE에 "LLM 인증 실패, key 갱신" 메타 알람.

sequenceDiagram
    autonumber
    participant SYS as 시스템
    participant LLM as 외부 LLM API
    participant OP as 운영자
    participant SRE as SRE
    participant MSG as 5채널(외부)
    SYS->>LLM: 초안 요청 (HTTPS)
    LLM-->>SYS: 인증 실패 / quota 초과
    Note over SYS: quota controller fail-open 판정 → SOP 원문 조회
    SYS->>MSG: SOP 원문 그대로 발송 ("AI 초안 없음" 표시)
    MSG->>OP: 운영자에게 정보 도달
    SYS->>SRE: 메타 알람 ("LLM 인증 실패, key 갱신")
    Note over SYS: 실패 사유 감사 기록

핵심 · AI 초안은 보너스, SOP는 본체. AI 실패가 대응을 멈추지 않음. 예외 · SOP 조회도 실패 시 원본 알람만+SRE critical / AI 광역 실패(분당 N회↑) 시 플랫폼팀 동시 알람.

운영자의 핵심 책임(진짜 알람인지·조치가 맞는지 판단)은 어느 시나리오에서도 시스템이 대체하지 않습니다.
